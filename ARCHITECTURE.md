# Multi-Device Report Sync Architecture

## Overview
This document describes the architecture for synchronizing and displaying reports from multiple devices running the app on the same local network. The goal is for every device to see all reports generated by any device, in real time or near real time, with no manual configuration.

## 1. Device Discovery ✅ **IMPLEMENTED**
- **mDNS/NSD**: Network Service Discovery with service type `_mrpreport._tcp`
- **Real-time Discovery**: Aggressive 5-second cleanup, immediate appearance/disappearance
- **Android 15 Compatible**: Version-aware service naming and NSD bug recovery
- **Persistent UUIDs**: Each device maintains consistent identity across app sessions
- **No Manual Config**: Fully automatic discovery on same WiFi network

## 2. Peer-to-Peer Sync ✅ **IMPLEMENTED**
- **HTTP Server**: Every device runs NanoHTTPD server to share its reports via REST API
- **Auto-Port Assignment**: Server auto-assigns available port, registered with mDNS for discovery
- **REST Endpoints**:
  - `GET /reports` - Returns all reports from this device
  - `GET /reports/since/{id}` - Returns reports newer than specified ID (incremental sync)
  - `GET /health` - Server health check
- **Integration**: HTTP server lifecycle tied to NSD service registration
- Each report is treated as a unique record, even if content is identical (no deduplication for now).

## 3. Report Exchange 🔄 **IN PROGRESS**
- **Phase 1 Complete**: HTTP server serves reports via REST API
- **Phase 2 Next**: HTTP client fetches reports from discovered devices
- **Incremental Sync**: Support for fetching only new reports since last sync
- **Automatic Merging**: Reports merged and stored locally on each device

## 4. UI
- The Reports tab shows all reports from all devices, grouped by device or time.

## 5. Handling Network Changes
- Devices periodically re-discover peers and re-sync, so new devices joining or leaving is handled automatically.

## 6. Conflict/Duplicate Handling
- For now, no duplicate filtering or conflict resolution is implemented. Every report entry, even if identical, is treated as a unique record. This ensures legitimate repeated entries are always shown.

## Advantages
- No single point of failure (no "main" device needed).
- All devices always have the full report list.
- Works even if some devices go offline temporarily.

## Technical Implementation

### Current Architecture
```
[Device A]                    [Device B]
├── NSD Service              ├── NSD Service  
├── HTTP Server :auto        ├── HTTP Server :auto
├── Report Database          ├── Report Database
└── Discovery Client         └── Discovery Client
```

### API Endpoints (Implemented)
- `GET /reports` - All reports from device
- `GET /reports/since/{id}` - Incremental sync
- `GET /health` - Server status

### Next Implementation: HTTP Client
```
When Device A discovers Device B:
1. Extract IP:Port from NSD service info
2. HTTP GET http://DeviceB:port/reports
3. Merge received reports into local database
4. Update UI to show combined reports
```
